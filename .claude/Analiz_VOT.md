# CVM — анализ скорости VOT и план ускорения

> Сравнение архитектуры ClaudeVoiceMaster (CVM) с проектом
> `ilyhalight/voice-over-translation` (VOT) и выводы о том, как приблизить
> CVM к ощущению мгновенной работы.

---

## 1. Главный вывод (одной фразой)

Скорость VOT — это **не быстрый код озвучки, а архитектурный сдвиг**: тяжёлая
работа сделана заранее, один раз, на сервере, и переиспользуется через
глобальный кэш. Клиент только скачивает готовый результат.

VOT **не переводит и не озвучивает видео сам**. Это тонкий клиент к закрытому
API Яндекса (тому, что работает в Яндекс.Браузере). Скачивание аудио,
распознавание речи (ASR), перевод, синтез голоса и сведение дорожки —
всё происходит **на серверах Яндекса**, а расширение получает готовый файл.

---

## 2. Сравнение архитектур

### CVM (текущая)
```
субтитры → перевод по кускам (Claude API) → озвучка по кускам (Edge TTS)
        → клиент сам собирает и синхронизирует дорожку в браузере
```
Десятки сетевых запросов, оркестрация живёт в браузере, всё в реальном времени.

### VOT
```
1 запрос «переведи это видео» → серверы Яндекса делают всё
        → возвращается ОДИН готовый MP3 на весь ролик (ссылка на S3)
        → + готовый JSON с субтитрами (тоже файл на S3)
```
По сути клиент сравнивает полноценный конвейер обработки с простым
скачиванием готового файла.

---

## 3. Откуда у VOT скорость — 3 настоящие причины

**1. Глобальный серверный кэш (главное).**
Первый перевод свежего, никем не тронутого видео у VOT тоже **не мгновенный** —
API возвращает статус `WORK_IN_PROGRESS` с полем `remainingTime`, и клиент ждёт
(секунды или минуты). Но как только *хоть один человек в мире* перевёл видео,
результат лежит в кэше Яндекса по ключу = ID видео. Все последующие зрители
сразу получают `SUCCESS` со ссылкой на готовый файл. Популярные ролики уже
переведены до тебя — отсюда ощущение мгновенности.

**2. Один артефакт вместо сотни запросов.**
Клиент делает ~1 запрос на перевод и ~1 на субтитры. Никакого
«перевести фразу → озвучить фразу → следующая». Возвращается цельный MP3 на
весь ролик на быстром объектном хранилище (`vtrans.s3-private.mds.yandex.net`),
браузер просто стримит его.

**3. Асинхронная модель «задача + поллинг».**
Клиент кидает задачу и опрашивает статус, не вися в ожидании:
```ts
const result = await client.translate({ originalUrl, originalLanguage, translationLanguage });
if (result.status === 'WORK_IN_PROGRESS') { result.remainingTime }
if (result.status === 'SUCCESS')          { result.translation.content.url } // готовый mp3
```
Технически запросы подписываются (protobuf + HMAC-подпись секретом из
Яндекс.Браузера), а прокси-воркеры (`vot-worker`, `media-proxy`, `vot-backend`)
нужны только для обхода блокировок — это инфраструктура доступа, а не перевода.

## 4. Рычаги для CVM (в порядке убывания эффекта)

### Рычаг №1 — общий серверный кэш (огромный выигрыш)
Сейчас кэш локальный (`cvm_cap_*`, `cvm_v_*`). Расширить Cloudflare Worker до
хранилища готовых результатов с ключом `videoId + targetLang + voice`
(KV / R2). Тогда если *любой* пользователь CVM уже смотрел ролик с такими
параметрами — следующий получит готовую озвучку мгновенно. Это ровно то,
что делает VOT. Worker уже есть в проекте — нужно добавить ему хранилище.

### Рычаг №2 — батч вместо поштучных запросов
Переводить **весь транскрипт за один запрос** к Claude (или несколько крупных),
а не фразу за фразой. Один большой запрос почти всегда быстрее, чем N маленьких,
из-за накладных расходов на каждый round-trip. Аналогично — синтезировать TTS
крупными логическими кусками (абзац / группа фраз), а не на каждую микро-реплику;
заодно уйдёт часть щелчков и пауз.

### Рычаг №3 — асинхронный pre-render до Play
Запускать перевод+озвучку фоновой задачей в момент **открытия** страницы видео,
а не по нажатию кнопки. В README уже заявлено «перевод заранее» — довести это до
полноценной модели «задача + статус» (по образцу `WORK_IN_PROGRESS` /
`remainingTime` / `SUCCESS`), чтобы к моменту просмотра дорожка уже была готова
и закэширована.

### Рычаг №4 — параллельный prefetch с буфером
Не сериализовать цепочку `перевод → tts → проигрывание` по каждой реплике.
Перевести всё разом, потом синтезировать ближайшие N сегментов параллельно,
опережая позицию плеера на буфер в несколько фраз. Воспроизведение никогда
не упирается в сетевой запрос.

### Рычаг №5 — отдавать готовые файлы по URL
Хранить синтезированные MP3 в хранилище Worker'а (R2/KV) и отдавать клиенту
ссылку, как Яндекс отдаёт S3-URL. Браузеру останется стримить, а не собирать
аудио в памяти на лету.

---

## 5. Реалистичная цель

«Всегда мгновенно как у VOT» без использования чужого готового конвейера
недостижимо: Claude (платный за токены) + Edge TTS по природе медленнее
пред-запечённой дорожки, а бесплатный глобальный кэш на всех так дёшево не
раздать. Реалистичная и честная цель:

> **Мгновенно при повторном просмотре + готово ещё до нажатия Play при первом.**

Переход с модели «оркестрация в браузере по кускам в реальном времени» на
модель «фоновая задача → готовый артефакт → общий кэш на Worker» закроет ~80%
разрыва в ощущении скорости, не отказываясь от главного преимущества CVM —
качества перевода через Claude и гибкости голосов.

---

## 6. Важно: про доступ к API Яндекса и юридическую сторону

- **Официального публичного API для этого у Яндекса нет.** VOT работает за счёт
  реверс-инжиниринга приватного API Яндекс.Браузера: извлечён протокол и
  секрет подписи запросов, расширение «представляется» Яндекс.Браузером.
- **Бесплатно — потому что Яндекс даёт перевод видео бесплатно своим
  пользователям браузера** (фича для привлечения аудитории). VOT
  «подсаживается» на этот бесплатный для пользователей сервис. Расходы несёт
  Яндекс.
- **Это почти наверняка нарушает условия использования Яндекса.** Сам клиент
  vtrans прямо пишет, что API «НЕ ПРЕДНАЗНАЧЕН для использования в ПО, кроме
  yandex.browser».
- **Для CVM копировать этот подход рискованно:** наследуются все риски
  (блокировка, смена секрета/протокола, юридическая серость, хрупкость).
  CVM уже использует легальные платные API (Claude, Edge TTS) — это правильная
  модель. Ускорение нужно брать из **архитектуры** (кэш, батч, async),
  а не из паразитирования на инфраструктуре Яндекса.

> Примечание: это не юридическая консультация. Для точной оценки рисков в твоей
> юрисдикции нужен профильный юрист.
