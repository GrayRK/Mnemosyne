## Стадия 1 — Popup + Live State Inspector

**Статус**: ✅ Выполнена
**Цель**: рабочий интерфейс с мгновенным откликом и инструментом визуального
тестирования.

### Задача

**1.1 — Popup (`entrypoints/popup/`).** 
Элементы управления:
- Поле ввода API-ключа (сохраняется в `chrome.storage.local`)
- Выбор языка перевода (русский по умолчанию)
- Выбор голоса TTS (список фильтруется по выбранному языку)
- Toggle: озвучка вкл/выкл
- Toggle: субтитры вкл/выкл
- Слайдер: громкость TTS (0–100%)
- Слайдер: приглушение оригинала (0–80%)
- Индикатор статуса («Готов» / «Переводим…» / «Ошибка»)
- Кнопка открыть Developer Tools (`</>`)

Требование: все изменения применяются немедленно, без перезагрузки страницы.

**1.2 — Интерфейс на странице с видео.**
- Кнопка «Перевести и озвучить» (запускает процесс)
- Субтитры внизу видео: белый текст на полупрозрачном фоне, скруглённые углы.

**1.3 — Live State Inspector (`entrypoints/inspector/`).**
Отдельная страница расширения, открывается кнопкой из popup. Показывает все
переменные состояния в реальном времени:

```
● videoVolume          = 0.4        Number    chrome.storage
● translationVolume    = 0.9        Number    chrome.storage
● subtitlesEnabled     = true       Boolean   chrome.storage
● ttsEnabled           = true       Boolean   chrome.storage
● selectedVoice        = "Dmitry"   String    chrome.storage
● targetLanguage       = "ru"       String    chrome.storage
● currentVideoId       = "dQw4w..."  String    runtime state
● translationStatus    = "ready"    String    runtime state
```

Обновляется через долгоживущее соединение. Каждое изменённое значение
подсвечивается на 1 секунду.

### Ход выполнения

**Каркас (отступления от исходной структуры, согласованы с пользователем):**
- `entrypoints/devtools/` → **`entrypoints/inspector/`**: имя `devtools` в WXT
  зарезервировано под настоящую панель Chrome DevTools (`devtools_page`),
  а нам нужна отдельная страница расширения, открываемая из popup.
- `content_bridge.ts` → **`content-bridge.content.ts`**: WXT распознаёт
  контент-скрипты только по шаблону `*.content.ts`.
- Добавлена папка **`lib/`** (`constants`, `types`, `storage`, `messaging`) —
  общий фундамент без магических констант и `any`.
- `wxt.config.ts`: `imports: false` — явные импорты из `#imports`, без авто-импортов.
- Везде унифицированный `browser` из `#imports` вместо глобального `chrome`.

**Popup:**
- Все настройки — в `chrome.storage.local` через типизированные
  `storage.defineItem` (`lib/storage.ts`); применяются мгновенно.
- Языки — короткий список (`ru, en, es, de, fr, zh, ja`).
- Голоса TTS читаются из `speechSynthesis.getVoices()` и фильтруются по языку;
  при смене языка список пересобирается, невалидный голос сбрасывается.
- API-ключ сохраняется с debounce; слайдеры хранятся как 0..1, отображаются в %.
- Добавлены (по запросу) два флага, по умолчанию выключены:
  `useYoutubeTranslation` (брать готовый перевод YouTube вместо API) и
  `autoStart` (запуск перевода при открытии). Логика — Стадия 2.

**Виджет на странице видео:**
- Кнопка вкл/выкл перевода в **Shadow DOM** (изоляция от стилей YouTube),
  монтируется поверх плеера (верх справа), уходит в фуллскрин с видео.
- Устойчивость к YouTube-SPA: монтаж по `yt-navigate-finish` + поллинг,
  таймеры/слушатели на `ctx` WXT (автоочистка при HMR).
- При SPA-навигации (другое видео / «назад») состояние сбрасывается в `false`.
- Контейнер субтитров **отложен до Стадии 2** (внешний вид настраивается на
  реальном тексте; пересекается со Стадией 1.1).

**Live State Inspector:**
- Отдельная страница `inspector.html`, открывается кнопкой `</>` из popup.
- Storage-значения обновляются через `storage.watch`; рантайм-значения
  (`translationActive`, `currentVideoId`, `translationStatus`) — через
  long-lived port из `background.ts`. Изменения подсвечиваются на 1 сек.
- API-ключ показывается замаскированным (не светим секрет).

**Известное ограничение (закрыто в Стадии 3.4):**
- Рантайм-состояние в `background.ts` было глобальным (одно на всё расширение).
  Для нескольких вкладок с видео нужно хранение per-tab — сделано в Стадии 3.4.

## Стадия 2 — Текст из видео + меню кэша в Inspector

**Статус**: ✅ Выполнена
**Цель**: получение, кеширование и отображение в Inspector текста из видео.

### Задача

**2.1 — Получение текста.**
- Получаем текст с YouTube-видео на оригинальном языке.
- Получаем текст автоперевода YouTube (по сути Google-перевод).

**2.2 — Кеширование.**
- Кешируем оба текста. Если текст уже в кэше — операции по извлечению из видео
  не выполняем.

**2.3 — Меню кэша в Inspector.** Дополнительный раздел в Inspector:
- Раскрывающийся список позиций: название, ссылка на видео, время создания.
- Каждая позиция раскрывается в три баннера: оригинальный текст, автоперевод,
  перевод с API (логика API — Стадия 3, пока заготовка).
- Сортировка списка по времени создания.
- При раскрытии позиции предыдущая раскрытая сворачивается (чтобы не
  загромождать пространство).
- Кнопка очистки всех кэшированных текстов.

**Примечание**: старт работы завязан на кнопку вкл/выкл перевода
(реализована в Стадии 1), а не на старт страницы.

### Ход выполнения

**Источник текста — ПЕРЕХВАТ запроса плеера (а не прямой fetch):**
- Сначала пробовали прямой `fetch` по `captionTracks[].baseUrl` из
  `movie_player.getPlayerResponse()` — YouTube отдавал **200 + 0 байт**:
  у этих URL нет `pot` (proof-of-origin token), который веб-клиент обязан
  прикладывать. Подделать его нельзя.
- Итог: мост (`content-bridge`, MAIN) патчит `window.fetch` и
  `XMLHttpRequest` на `document_start` и перехватывает **реальный запрос
  плеера** к `/api/timedtext` (там валидный `pot`) — берёт оттуда готовое
  тело оригинала. Перехват один на страницу (guard-флаг на `window`).
- Автоперевод YouTube (Google): к перехваченному URL добавляем
  `&tlang=<язык>&fmt=json3` — `pot` тот же, запрос отрабатывает.
- Парсинг с автоопределением формата: `json3` (тайминги `tStartMs`/
  `dDurationMs` — задел под озвучку Стадии 4) с фолбэком на «сырой» XML.

**Авто-включение субтитров (чтобы было что перехватывать):**
- По запросу из content мост: если CC выключены — включает их (API плеера
  `setOption('captions',…)`, иначе **клик по родной кнопке** `.ytp-subtitles-button`),
  ждёт перехвата (поллинг до 6 с), затем **возвращает как было** (выключает,
  если включал он). На время операции прячет контейнер субтитров CSS-ом —
  на экране ничего не мелькает.

**Каналы связи:**
- Мост(MAIN) ↔ content(ISOLATED): `window.postMessage`, конверт
  `{ source: 'cvm-bridge', message }`; одноразовый `request-extraction` →
  `extraction-data` (videoId, title, язык, перехваченные URL+тело) с таймаутом.
- content → background: `runtime.sendMessage` с ответом — `cache-lookup`
  (есть ли запись / есть ли перевод на язык) и `cache-store` (запись).
- Inspector ↔ background: long-lived порт; `runtime-state` / `cache-list` /
  `cache-entry` (вниз) и команды `request-cache-list` / `request-cache-entry`
  / `clear-cache` (вверх).

**Модель кэша (`lib/cache.ts`) — ОДНА запись на видео:**
- Ключ `cvm_cap_{videoId}` (без языка) + лёгкий индекс `cvm_cap_index`.
  Внутри записи: оригинал один раз, словарь автопереводов
  `translations: { ru: [...], es: [...] }`, заготовка `apiTranslation: null`.
- Префикс `cvm_cap_` отделён от зарезервированного под Стадию 3 ключа API
  `cvm_v_{videoId}_{lang}`.
- Смена целевого языка **не плодит запись**, а добавляет язык
  (`upsertEntry`). Если язык уже есть — извлечение пропускается.
- Если целевой язык == оригиналу (по базовому коду, `en-US`≈`en`) —
  автоперевод не нужен.
- `readIndex` устойчив к повреждённым данным (отбрасывает невалидные
  элементы) — иначе битый индекс ронял запись и «Очистить кэш».

**Флаг «Использовать автоперевод» (`useYoutubeTranslation`):**
- Включён → кэшируем оригинал + автоперевод на целевой язык.
- Выключен → кэшируем **только оригинал** (+ заглушка API); автоперевод не
  тянется (оригинал нужен Стадии 3 для перевода через API).

**Авто-синхронизация при активном виджете:**
- Пока виджет включён, `storage.watch` на `targetLanguage` и
  `useYoutubeTranslation` сам до-кэширует новый язык — без передёргивания
  кнопки. Запуски пайплайна сериализованы (без наложения; перезапуск с
  актуальными настройками).

**Автозапуск (`autoStart`):**
- При открытии `/watch` и при SPA-навигации на новое видео виджет сам
  переходит в «включено» и запускает пайплайн (если флаг включён).

**Меню кэша в Inspector:**
- Аккордеон под таблицей: позиции (название, ссылка `↗`, время), раскрытие
  одной сворачивает предыдущую (видимость через класс `.open`, не атрибут
  `hidden`). Внутри — баннеры: оригинал + по одному на каждый язык +
  заглушка API.
- Кнопка-переключатель сортировки по времени создания (новые↔старые).
- Полный текст позиции подгружается при раскрытии (`request-cache-entry`),
  список обновляется живо. Кнопка «Очистить кэш» (с подтверждением).

**Окружение dev (`wxt.config.ts`):**
- Постоянный профиль Chrome (`~/.cvm-chrome-profile`, `keepProfileChanges`) —
  cookie-согласие/сессия YouTube сохраняются между запусками (иначе чистый
  профиль не отдаёт субтитры). Папка ВНЕ проекта — иначе Vite-вотчер падает
  на залоченных Chrome-файлах.
- Флаг `--disable-blink-features=AutomationControlled` — маскировка
  автоматизации, YouTube отдаёт обычную сессию.

## Стадия 3 — Claude API + API Monitor + калькулятор стоимости

**Статус**: ✅ Выполнена (ядро/монитор/калькулятор; доработки скорости,
per-tab и виджет стоимости — в Стадии 3.4).
**Цель**: рабочий перевод субтитров с полной прозрачностью работы API.

### Задача

**3.1 — Интеграция API (`background.ts`).**
- Батчинг субтитров (размер подбирается в Стадии 3.4).
- Параллельные батчи (пул воркеров).
- Кэш: `chrome.storage.local`, ключ `cvm_v_{videoId}_{lang}`.
- Retry-логика: повтор при сетевой ошибке.
- Строгий системный промпт (без артефактов).
- На время перевода — индикатор прогресса (`N/M` батчей).

**3.2 — API Monitor (расширение Inspector).**
Внутри позиции кэша (см. примечание Стадии 2), у баннера API — кнопка,
вызывающая модалку с информацией:

```
* Используемая модель Claude API - [Sonnet 4.6]
* Кол-во сформированных батчей - [3]
* Кол-во символов на батч - [510]
* Время на 1 батч - [2s]
* Время на весь текст - [6s]
```

**3.3 — Калькулятор стоимости.**
- Точный калькулятор на основе реальных замеров (токены, стоимость, время).
- Стоимость за запрос уточняется у пользователя / калибруется.
- Калькулятор размещён прямо в Inspector.
- Совместно с пользователем довести скорость запросов до приемлемой.

- **3.4.1 — Оптимизация скорости перевода через API.** Подобрать размер батча
  (`API_BATCH_SIZE`) и параллелизм (`API_BATCH_CONCURRENCY`), оценить узкие
  места (латентность модели, размер ответа, `max_tokens`), довести до
  приемлемой скорости.
- **3.4.2 — Точность метрик API Monitor.** Проверить учёт токенов (`usage`
  per-batch vs суммарный), «символов на батч» (средн. vs факт), тайминги
  (wall-clock vs сумма per-batch при параллелизме), `videoSeconds`.
- **3.4.3 — Корректность калькулятора.** Согласованность базы (символы) между
  фиксацией и расчётом, формулы `R = $/символ` и `D = символов/мин`, усреднение
  по выборкам, клампы ползунков, оценка токенов (`tokensPerChar`).
- **3.4.4 — Рантайм-состояние per-tab** (перенесено из Стадии 3):
  `Map<tabId, state>`, очистка по `chrome.tabs.onRemoved`, изоляция пайплайна
  по вкладке, выбор/показ вкладки в Inspector.

### Ход выполнения

**3.1 — Ядро перевода (`background.ts`):**
- Перевод запускается в ветке `useYoutubeTranslation = OFF` пайплайна
  (`content.ts`): ON → автоперевод YouTube (Стадия 2), OFF → Claude API.
- Запрос к `/v1/messages` (`claude-sonnet-4-6`), заголовки `x-api-key` +
  `anthropic-version` + `anthropic-dangerous-direct-browser-access`.
  Для fetch из фона добавлен `host_permissions: https://aiprimetech.io/*`.
- Батчинг (см. 3.4 про размер), пул на **3** воркера, `withRetry` — 1 повтор
  ТОЛЬКО на сетевые/HTTP-ошибки (не на формат).
- **Нумерованный протокол вместо строгого JSON.** Реплики уходят как
  `[1] текст`, модель возвращает `[n] перевод`. Сборка по номерам: потерянный
  номер → подставляется оригинал. Длина результата всегда = числу сегментов
  (тайминги 1:1 для Стадии 4), несовпадение количества НЕ рушит батч и не
  плодит retry-шторм. Решило баг «длина 128 ≠ 150» и «8 запросов на 1/4».
- Хранение per-language: `apiTranslations: Record<lang, CaptionSegment[]>`
  (по аналогии с `translations`), в мете `apiLanguages[]`.
- Индикатор `N/M` на кнопке виджета (фон шлёт прогресс в tab через
  `tabs.sendMessage`) + рантайм-строка `translationProgress` в Inspector.

**3.2 — API Monitor (Inspector):**
- Метрики собираются по ходу перевода и пишутся в запись кэша per-language:
  `apiMeta: Record<lang, ApiTranslationMeta>` (модель, число батчей, символы,
  токены из `usage`, тайминги per-batch и общий, `videoSeconds`). Переживают
  перезапуск SW.
- У баннера «Перевод Claude API · {язык}» — кнопка **📊 Монитор** → модалка с
  таблицей метрик (закрытие: крестик / фон / Esc).
- ⚠️ Точность метрик доработана в Стадии 3.4.

**3.3 — Калькулятор стоимости:**
- Эмпирическая калибровка вместо хардкода цен: калькулятор учит `R = $/символ`
  и `D = символов/мин` усреднением по выборкам ТЕКУЩЕЙ модели (база — символы
  оригинала). Цены за токены у пользователя НЕ запрашиваются.
- Фиксация в модалке монитора: поле `$` + «Зафиксировать» → валидация →
  `confirm` → блокировка поля. Стоимость пишется и в `apiMeta[lang].costUsd`,
  и в отдельное хранилище `cvm_cost_samples` (переживает «Очистить кэш»).
  Фиксацию выполняет фон (`record-api-cost`), сам берёт объём/токены/длительность
  из `apiMeta`. Повторная фиксация запрещена (`already-fixed`).
- Калькулятор на странице Inspector: три зависимых контрола (длина видео —
  ползунок; символы и стоимость — точный ввод), связаны через `R`/`D`; живут
  через `storage.watch` на выборках. Кнопка «Сбросить калибровку».
- Тема — тёмная, в стиле Inspector.
- ⚠️ Неправдоподобность результата исправлена в Стадии 3.4.

**3.4.1 — Скорость перевода (пошаговые замеры на эталонном видео):**
- `API_BATCH_CONCURRENCY` 3 → **6**: total 75с → 40с. **6 = реальный потолок Chrome**
  (~6 соединений на хост); выше — лишние запросы висят в очереди браузера и только
  замедляют (батч25/conc12 давал 56с при быстром сервере).
- Размер батча: **серверная латентность от размера батча НЕ зависит** (зомби случается
  и на 25 строках). Крупные батчи (`API_BATCH_SIZE = 50`) → меньше запросов = меньше
  «бросков кубика» на зомби + одна волна → устойчивее мелких (25/conc6 = 64с). Итог:
  **батч 50 / concurrency 6**.
- **Главное узкое место — серверная tail-латентность прокси.** По CSV из ЛК
  `aiprimetech.io` держит отдельные запросы до **140–170с** (подтверждено колонкой
  Duration самого сервера, не артефакт очереди). Клиентом не лечится: total пляшет
  ~27–73с в зависимости от чужой нагрузки на shared-прокси.
- **Backoff на 429/перегрузку** (статусы 429/529/503): экспонента + джиттер, чтобы
  параллельные воркеры не били синхронно. До этого retry был без задержки — на
  rate-limit бесполезен.
- **Адаптивный таймаут батча**: `max(45с, 2.5 × медиана латентности батчей текущего
  прогона)` → abort + перезапрос. Пороги length-инвариантны (медиана = от 50-строчных
  батчей, не от длины видео), поэтому корректны и для 2-часовых плотных роликов.
- **Хеджирование (дубль-запрос) пробовали и ОТКАЗАЛИСЬ.** На queue-bound прокси дубль
  добавляет нагрузку (8 запросов вместо 6 → 52с против 27с) и сам зомбирует (в CSV
  виден дубль на 169с). Оставили только адаптивный таймаут.
- **Потоковую запись в кэш на каждый батч пробовали и ОТКАТИЛИ.** Переписывание всей
  записи (283 сегмента) в `chrome.storage` ×N грузило service worker и расходило
  клиент с сервером (+~30с к total). Диагностика (запись только в конце) вернула 27с.
  Пишем финально один раз; прогрессивную отдачу перенесли в Стадию 4 (лёгкие сообщения
  консьюмеру TTS, без storage на каждый батч). Поля `completedBatches`/`complete` в
  `apiMeta` оставлены как задел; `cache-lookup` теперь требует `complete === true`.
- Временная инструментация `[CVM speed]` в логах service worker использовалась для
  замеров и **удалена** по завершении тюнинга.

**Кросс-загрязнение кэша между видео (баг Стадии 2, найден и исправлен):**
- Перехват субтитров в мосте хранился в модульных переменных **без привязки к видео**;

  `buildExtractionData` никогда не отдаёт тело от чужого видео.

**3.4.3 — Виджет стоимости + калькулятор:**

## Стадия 4 — TTS

**Цель**: озвучка переведённых субтитров синхронно с видео, с нейронным голосом и
работой «из коробки» (без настройки и VPN у конечного пользователя).

**Статус**: ✅ Базовый функционал готов и работает: перевод + нейронная
озвучка Edge с живой синхронизацией темпа + караоке-субтитры.

### Ход выполнения

**Доставка текста:** `get-translation` (messaging + background) — content забирает
финальные переведённые сегменты из кэша (`apiTranslations[lang]` или
`translations[lang]` по флагу `useYoutubeTranslation`).

**Абстракция движка (`lib/tts.ts`):** интерфейс `TtsEngine`
(`speak / prefetch / clearCache / cancel / pause / resume / isSpeaking`). Реализации:
- `WebSpeechEngine` — системные голоса браузера (`speechSynthesis`).
- `EdgeTtsEngine` — нейронные голоса Microsoft Edge, синтез на стороне прокси,
  проигрывание MP3 через `<audio>`; есть префетч-кэш и fallback на Web Speech при
  сбое.
- `CompositeTtsEngine` — маршрутизатор: держит обе реализации, переключается на лету
  (`setEngine`). Фабрика `createTtsEngine()` возвращает его.

**Два движка, выбор в popup:** «Движок озвучки» (Edge / Web Speech). Список голосов
зависит от движка и языка: Edge — каталог нейронных голосов (`EDGE_VOICE_CATALOG`,
ru/en/es/de/fr/zh/ja, несколько на язык), Web Speech — системные голоса по языку.
Выбранный голос хранится отдельно для каждого движка (`selectedVoice` /
`selectedVoiceEdge`). Целевой язык передаётся в синтез; смена языка меняет голоса.

**Обход региональной блокировки Edge (главное достижение):** прямой эндпоинт
Microsoft из браузера недоступен (нельзя задать Origin/User-Agent + DPI-блок в РФ).
Решение — общий прокси **Cloudflare Worker** (`worker/edge-tts-worker.js`, задеплоен
под аккаунтом владельца). Worker вне РФ, ставит нужные заголовки и повторяет протокол
`edge-tts` (токен Sec-MS-GEC, ConnectionId, версия Chromium). URL вшит как
`DEFAULT_TTS_ENDPOINT` — пользователю настройка не нужна. Эндпоинт переопределяется
в Inspector (`cvm_tts_endpoint`). `tools/edge-tts-relay.py` (локальный Python-хелпер
через VPN) — только для отладки. Подробности: память проекта
`tts-edge-via-cloudflare-worker`.

**Планировщик синхронизации (`content.ts`):** слушает `timeupdate`, озвучивает реплики
строго последовательно (без наложения), реплики не пропускает. Цепочка реплик
запускается сразу по `onEnd` (без простоя на `timeupdate`). Префетч следующей реплики
во время текущей (по `cacheId`, сессия в префиксе). Темп считается «по месту» на
штатный старт реплики и кэшируется (`planRate`) — один и тот же для префетча и
воспроизведения; зажат в пользовательский диапазон.

**Регулировки (popup + Inspector):**
- Диапазон скорости: «Скорость от» (база) / «Скорость до» (потолок), `×1..×10`,
  масштабируются `playbackRate`. Живо.
- Сдвиг времени озвучки/субтитров (`ttsOffsetMs`, ±3000 мс) — подгонка синхрона. - Лишнее
- Реакция на seek / pause / ratechange (пауза замораживает речь без перечитывания).
- Приглушение оригинала (ducking): мгновенно при старте, плавное восстановление 300 мс.
- Живой учёт смены движка / голоса / диапазона / громкости / `ttsEnabled` без
  перезапуска.

**Караоке-субтитры:** оверлей плеера — предыдущая / текущая (ярко, с множителем темпа
`×N`) / следующая строки. Тоггл `subtitlesEnabled`. Физическое вертикальное положение
настраивается (`subsPositionPct`, 0 верх … 100 низ).

**Inspector:** раздел «Озвучка (TTS)» с подразделами (Темп речи / Синхронизация /
Edge-эндпоинт), строки состояния всех настроек.

**Глубокий префетч (скорость).** Префетчили только следующую реплику
(`prefetchSegment(index + 1)`) — на плотной речи один запрос наперёд не успевал спрятать
RTT к Worker. Теперь синтезируем `TTS_PREFETCH_AHEAD` (=6) реплик вперёд; ёмкость
префетч-кэша Edge поднята до `EDGE_TTS_PREFETCH_CACHE_LIMIT` (=10), чтобы окно не
вытеснялось. Префетч мемоизирован по `cacheId`, повторные запросы дёшевы.

**Темп не реагировал на отставание (разгон не работал).** Корень — конфликт
глубокого префетча и адаптации: Edge ВШИВАЕТ темп в MP3, поэтому `planRate` запекался за
1–6 реплик до воспроизведения и кэшировался. Любая модель, считающая темп заранее (и штатный
`seg.start`, и прогноз `projectedStartMs` из первой попытки 4.x), оптимистична — при базовом
темпе «речь укладывается в окно» → `min`, а реальное отставание копится УЖЕ ПОСЛЕ запекания и
закэшированный rate его не видит. Отсюда «в начале поймали ускорение, потом упёрлись в min».
Решение — **разделить запечённый темп и живую коррекцию**: `planRate` остаётся baseline для
синтеза/префетча (считается на реальный `videoNowMs()` при первом префетче), а в момент СТАРТА
реплики `speedAdjustFor(index) = rateFor(index, videoNowMs()) / planRate(index)` даёт множитель
под фактическое отставание. Применяется поверх готового аудио без пересинтеза: Edge —
`audio.playbackRate` (`preservesPitch=true`), Web Speech — множитель к `utterance.rate`. Это
замкнутая обратная связь по реальному `now`: отстаём → коррекция растёт → реплика звучит
быстрее → догоняем. Прогноз `projectedStartMs`/`plannedStart` из первой попытки удалён.

**Проскоки системного голоса при Edge (исправлен баг ниже).** `EdgeTtsEngine` на
сбое синтеза сразу читал реплику через Web Speech (`fallback.speak`) — отсюда чередование
нейронного и системного голоса. Теперь: синтез в `edge-tts.synthesize` идёт с ретраями
(`EDGE_TTS_SYNTH_RETRIES`=2, пауза `EDGE_TTS_SYNTH_RETRY_DELAY_MS`) — транзиентные сбои
Worker (холодный старт CF / таймаут) переживаются прозрачно; при окончательном провале
реплика **молча пропускается** (karaoke-субтитр остаётся), без подмены системным голосом.
Fallback-движок из `EdgeTtsEngine` удалён.
